AWS S3는 Signature Version 4 검증 시 요청 시간도 함께 확인합니다.
클라이언트 시간과 AWS 기준 시간 차이가 너무 크면 요청 자체를 거부합니다.

대표적으로 보게 되는 에러는 아래와 같습니다.

```text
RequestTimeTooSkewed
```

실무에서는 보통 "서버 시간이 조금 틀어졌네" 정도로 끝나지 않습니다.
S3 접근, presigned URL, 일부 인증 로직이 함께 실패하기 때문에 원인을 빨리 분리해야 합니다.

## 이번 장애를 다시 보며 남긴 메모

- 처음에는 SDK retry나 presigned URL 쪽 문제처럼 보여도, 결국 host 시간부터 확인하는 게 가장 빨랐다.
- 폐쇄망에서는 "NTP 동기화하세요"가 바로 답이 아니라 내부 시간 기준을 어떻게 둘지까지 같이 봐야 했다.
- Docker를 쓰고 있으면 컨테이너 설정부터 보기 쉬운데 실제로는 host 시간이 먼저인 경우가 많았다.

## 1. 왜 이 에러가 발생하나

AWS는 요청 서명에 포함된 시간을 기준으로 재생 공격을 방지합니다.
이 때문에 클라이언트 시간과 AWS 시간 차이가 대략 15분 이상 벌어지면 요청이 거부됩니다.

다만 운영 경험상 15분 직전까지 버티는 식으로 두면 안전하지 않습니다.
보통 5분 이상 차이가 나기 시작하면 인증 관련 이슈가 점점 생기기 쉬워집니다.
실제 운영 환경에서는 시간 drift를 1분 이내로 유지하는 편이 안전합니다.

## 2. 이번 환경에서 NTP가 바로 답이 아닌 이유

현재 조건은 다음과 같습니다.

- 폐쇄망
- 외부 NTP 서버 접근 불가
- 인터넷 시간 서버 접근 불가

즉 일반적으로 많이 말하는 "NTP 동기화하세요"가 당장 적용되지 않는 상황입니다.
이 경우 현실적으로 가능한 대응은 3가지 정도로 압축됩니다.

## 3. 대응 1: 서버 시간을 수동으로 맞춘다

가장 빠르게 문제를 끄는 방법입니다.
인터넷이 되는 PC에서 현재 UTC 시간을 확인한 뒤, 폐쇄망 서버 시간을 직접 맞춥니다.

예시:

```bash
date -u
sudo date -s "2026-03-09 11:45:00"
sudo date -u -s "2026-03-09 02:45:00"
date
```

핵심은 AWS가 보는 기준 시간과 크게 어긋나지 않게 만드는 것입니다.
운영 중인 장애를 즉시 멈춰야 할 때는 이 방식이 가장 단순하고 효과가 빠릅니다.

단점도 분명합니다.

- 시간이 지나면 drift가 다시 쌓인다
- 몇 주 뒤 같은 문제가 재발할 수 있다
- 여러 서버가 있으면 서버 간 시간이 또 어긋날 수 있다

즉 이 방법은 "즉시 복구"에는 좋지만 "운영 체계"로 보기에는 약합니다.

## 4. 대응 2: 폐쇄망 내부 NTP 서버를 둔다

운영 관점에서 가장 권장되는 방식입니다.
폐쇄망 안에 기준 시간 서버를 하나 두고, 나머지 서버들이 내부에서만 동기화하도록 구성합니다.

구조 예시는 아래처럼 생각하면 됩니다.

```text
인터넷 가능 기준 서버
  -> 내부 NTP 서버
    -> 개발 서버 / 배치 서버 / 서비스 서버
```

예를 들어 `chrony`를 쓴다면 내부 서버를 기준으로 이런 식의 구성이 가능합니다.

```conf
server time-server.local iburst
```

이 방식의 장점은 명확합니다.

- 운영 안정성이 높다
- 모든 서버 시간이 같은 기준을 본다
- AWS 인증, DB 시간 비교, 로그 상관관계 분석이 쉬워진다
- 사람이 수동으로 시간을 맞출 필요가 줄어든다

폐쇄망을 쓰는 금융, 제조, 공공 계열 환경에서도 결국 내부 시간 기준 서버를 두는 방식이 가장 일반적입니다.

## 5. 대응 3: 코드에서 clock skew 보정에 기대는 방법

이건 임시 우회책에 가깝습니다.
Python `boto3`와 `botocore`는 일정 수준의 clock skew 보정과 retry 동작을 가집니다.
그래도 서버 시간이 크게 틀어진 상태를 근본적으로 해결해주지는 못합니다.

예시:

```python
import boto3
from botocore.config import Config

config = Config(
    retries={
        "max_attempts": 10,
        "mode": "standard",
    }
)

s3 = boto3.resource("s3", config=config)
```

또는 세션 기반으로 클라이언트를 열 수도 있습니다.

```python
import boto3
import botocore

session = boto3.session.Session()
client = session.client(
    "s3",
    config=botocore.client.Config(
        retries={"max_attempts": 10}
    ),
)
```

이 접근의 한계:

- 서버 시간이 몇 시간 이상 틀어지면 보정만으로는 실패할 수 있다
- retry는 네트워크 일시 오류 완화에는 좋지만 잘못된 시스템 시간을 고치지는 못한다
- 장애 원인을 코드 레벨에서 숨겨버릴 수 있다

따라서 retry 설정은 보조 수단으로만 보는 편이 맞습니다.

## 6. Docker 환경이라면 Host 시간을 먼저 봐야 한다

Docker를 쓰면 컨테이너 시간만 따로 흐르는 것처럼 오해하기 쉽습니다.
실제론 대부분 Host 시간이 기준입니다.

즉 Host 시간이 틀리면 컨테이너 안의 애플리케이션도 같은 문제를 겪습니다.

확인 예시:

```bash
docker exec -it <container_name> date
date
```

컨테이너 안에서만 시간을 손보려 하기보다 Host time이 정상인지 먼저 보는 편이 빠릅니다.

## 7. 운영 우선순위는 이렇게 잡는 편이 현실적이다

장애 복구와 운영 안정성을 나눠서 보면 우선순위가 명확합니다.

1. Host 서버 시간을 수동 수정해 장애를 즉시 멈춘다
2. 폐쇄망 내부 NTP 서버를 구축해 재발을 막는다
3. SDK retry와 clock skew 보정은 보조 장치로 둔다

단기 처방과 장기 처방을 섞어 생각하면 대응이 느려집니다.
지금 장애가 난 상태라면 먼저 시간을 맞추고, 운영 체계는 그 다음 단계에서 정리하는 것이 효율적입니다.

## 8. 실무 체크리스트

- 서버 시간이 UTC 기준으로 크게 틀어지지 않았는가
- 여러 서버가 있다면 서로 같은 기준 시간을 보고 있는가
- Docker라면 Host 시간이 정상인가
- 애플리케이션 retry 설정만 믿고 있지 않은가
- presigned URL, S3 업로드/다운로드, 배치 작업이 함께 실패하는지 확인했는가

`RequestTimeTooSkewed`는 코드 버그처럼 보이지만 실제로는 운영 시간 관리 문제인 경우가 많습니다.
폐쇄망에서는 더더욱 "정확한 시간 기준을 어떻게 유지할 것인가"가 설계 항목에 들어가야 합니다.
이번 메모를 남기면서도 비슷한 장애가 다시 나오면 코드보다 먼저 UTC 기준 시간, host 시간, 내부 NTP 유무를 순서대로 확인해야겠다는 기준이 분명해졌습니다.
